================================================================================
ARCHITECTURE CHOICES
================================================================================

PATTERN: Clean Architecture (3 layers)

Presentation → Core → Data

Layers:
1. Presentation: UI, widgets, screens, animations
2. Core: Providers (state), themes, constants
3. Data: Models, repositories, services, storage

Benefits: Separation of concerns, testable, scalable, maintainable

STATE MANAGEMENT: Provider
Why: Official recommendation, simple API, performant, minimal boilerplate
Alternatives considered: Riverpod (complex), Bloc (verbose), GetX (less safe)

Providers:
• HealthProvider: Data, loading, search, filtering
• ThemeProvider: Theme mode, persistence

DATA PERSISTENCE: SharedPreferences
Why: Perfect for JSON, fast, simple, reliable
Alternatives: Hive (overkill), SQLite (too complex)

REPOSITORY PATTERN
Flow: UI → Provider → Repository → Service → Storage
Benefits: Abstraction, easy to swap sources, mockable, consistent errors

DEPENDENCY INJECTION
Simple constructor injection via Provider
HealthDataService → HealthRepository → HealthProvider
